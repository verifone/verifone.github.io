19 /**
20  * Example POS app for the Commerce Platform Payment API.
21  * This class is built as an example and is not compilable source code, but is meant to provide direction.
22  * It therefore favors simplicity and accessibility over Android best practices.
23  *
24  * In a typical payment flow, the steps are as follows:
25  * <ul>
26  * <li>A transaction session is opened, reserving the payment terminal for use by this app.
27  * {@code mTransactionManager.startSession(this); }</li>
28  * <li>In the Commerce Listener, wait for the Session Started or Session Resumed event before continuing. A session must be open to be able to do a transaction, or update the basket on the customer-facing screen, or perform other similar actions.
29  * {@code
30  * public CommerceEvent.Response handleEvent(CommerceEvent event)
31  * CommerceEvent.SESSION_STARTED
32  * CommerceEvent.SESSION_RESUMED
33  * }</li>
34  * <li>The basket is opened, allowing the app to add merchandise to the basket and update the customer-facing screen with the items and running total.
35  * {@code mBasketManager = mTransactionManager.getBasketManager(); }</li>
36  * <li>As items are entered added in the POS app, the items should be added to the basket which triggers the customer-facing display to update with the new item and the new running total.
37  * {@code mBasketManager.addMerchandise(merch); } </li>
38  * <li>Once all of the items have been added, we finalize the basket, allowing any other apps to perform any necessary modifications.
39  * {@code mBasketManager.finalizeBasket(); }</li>
40  * <li>Once the basket is finalized, we start the payment process.
41  * {@code
42  * AmountTotals currentTotals = mBasketManager.getCurrentAmountTotals();
43  * Payment payment = new Payment();
44  * payment.setRequestedPaymentAmount(currentTotals.getRunningTotal();
45  * mTransactionManager.startPayment(payment); }</li>
46  * <li>As payments are received, the {@link #handleEvent(CommerceEvent event) handleEvent(CommerceEvent event)} method is called with a {@link com.verifone.commerce.payment.PaymentCompletedEvent PaymentCompletedEvent} object, allowing the appropriate receipt to be printed.
47  * {@code
48  * PaymentCompletedEvent payEvent = (PaymentCompletedEvent) event;
49  * payEvent.getPayment().getReceipt().print(); }</li>
50  * <li>Finally, we end the session, completing the transaction.
51  * {@code mTransactionManager.endSession(); } </li>
52  * </ul>
53  * This description only covers a typical simple payment flow. More complicated flows can be triggered at various points in this process, but are described in more detail in the technical documentation.
54  *
55  */

91  /** The listener method. Allows handling of various session events. */
92  public CommerceEvent.Response handleEvent(CommerceEvent event) {
93  switch (event.getType()) {
94  case CommerceEvent.SESSION_STARTED:
95  case CommerceEvent.SESSION_RESUMED:
96  // Update the local variable to allow other actions to be performed.
97  mIsSessionActive = true;
98  /*
99  * Gets the Basket Manager and assigns it to a local variable for use later.
100  * {@code mBasketManager = mTransactionManager.getBasketManager();}
101  */
102  mBasketManager = mTransactionManager.getBasketManager();
103  break;
104  case CommerceEvent.SESSION_ENDED:
105  mIsSessionActive = false;
106  break;
107  case CommerceEvent.SESSION_ERROR:
108  // Should notify the user that something is wrong.
109  break;
110  case CommerceEvent.SESSION_END_FAILED:
111  // Should consider retrying to end the session, or maybe notify the user.
112  break;
113  case CommerceEvent.SESSION_CLOSED:
114  // Happens when the session is closed without having endSession() called.
115  case CommerceEvent.SESSION_START_FAILED:
116  case CommerceEvent.SESSION_RESUME_FAILED:
117  // Notify user of error using event.getMessage().
118  mIsSessionActive = false;
119  break;
120  case TransactionEvent.TRANSACTION_STARTED:
121  // Similar to TRANSACTION_PAYMENT_STARTED, but for voids, refunds, and other transaction types.
122  break;
123  case TransactionEvent.TRANSACTION_PAYMENT_STARTED:
124  // The payment has started. REQUEST events will follow allowing the POS to determine a method of payment, or allow the customer to select on the terminal.
125  break;
126  case TransactionEvent.TRANSACTION_PAYMENT_COMPLETED:
127  // Handle the payment completion, printing the receipt in this case.
128  PaymentCompletedEvent payEvent = (PaymentCompletedEvent) event;
129  payEvent.getPayment().getReceipt().print();
130  // Reset the session, since we're done with the current one and want to be ready for another cart/payment.
131  // In the case of split tender or partial auth, we would need to check the event for the actual amount paid,
132  // and start another payment for any remaining amount.
133  mTransactionManager.endSession();
134  mTransactionManager.startSession();
135  break;
136  case TransactionEvent.TRANSACTION_ENDED:
137  break;
138  case TransactionEvent.TRANSACTION_ERROR:
139  // Notify user of error using event.getMessage().
140  break;
141  case TransactionEvent.TRANSACTION_FAILED:
142  // Notify user of error using event.getMessage().
143  break;
144  case TransactionEvent.TRANSACTION_NOTIFICATION:
145  break;
146  case TransactionEvent.TRANSACTION_REQUEST:
147  break;
148  case BasketEvent.TYPE:
149  // These are received when items are successfully added, modified, or removed from the basket, or if an error occurred when updating the basket.
150  break;
151  default:
152  break;
153  }
154  return null;
155  }
