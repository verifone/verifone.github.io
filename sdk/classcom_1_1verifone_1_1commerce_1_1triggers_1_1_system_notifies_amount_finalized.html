Detailed Description

Fired by Payment Application when amount was received from POS or entered directly.

This informs the Commerce Applications on the final amount and intermediate amounts, if any.

This represents the sum of adjustments that have to be applied to the initial amount excluding the gratuity.

This amount may either be positive, negative, or zero. The adjustements may be done by POS or by CP applications. The value may therefore change throughout the transaction.

◆ getFinalAmount()

abstract BigDecimal getFinalAmount ( )
abstract

This represent the projected final amount to be paid.

The value may change throughout the transaction. It always obeys to the following formula:

FinalAmount = InitialAmount + AdjustedAmount + GratuityAmount - AlreadyApprovedAmount

The value of FinalAmount is used as initial Transaction Amount value for the payment phase.

◆ getInitialAmount()

This represents the amount that payment is attempted.

This is either captured by the payment application or by POS. It is always positive and non-zero. It does not include the gratuity.
